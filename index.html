03 The Secret Sauce
How the Prompt Gets Assembled
Before the AI thinks, OpenClaw packs everything it needs into a single prompt — like preparing a briefcase before a meeting.
Build Steps
1. System Prompt
Base instructions — who the agent is, safety rules, personality.
2. Memory Context
MEMORY.md + daily logs + hybrid search results from indexed files.
3. Session Skills
Instructions from SKILL.md files that passed requirement checks at session start.
4. Conversation History
Recent messages, trimmed to fit. Older messages are summarized.
5. User Message
The actual request — placed last so the agent treats it as the primary instruction.

05 How the Agent Gets Things Done

      
CLI vs MCP — Why It Matters

      
When OpenClaw's agent needs to interact with external systems — JIRA, GitHub, databases — it has two ways to do it. The choice has a massive impact on cost, speed, and reliability.

    


    
    

      
      

        

        

          
💻

          

            
CLI Approach

            
What OpenClaw uses by default

          

          
Recommended

        

        
The agent runs the same commands a developer would type in a terminal — gh pr list, aws s3 ls, kubectl get pods. Direct, fast, no middleman.


        
        

          
terminal

          

            $ gh pr list --state open --json title

            → [{title: "Fix auth flow"}, ...]

            $ gh pr view 412 --json reviews

            → 2 approvals, 0 changes requested
          

        


        

          
✓10-32x cheaper per operation

          
✓100% reliability in benchmarks

          
✓Minimal tokens — near-zero overhead

          
✓Battle-tested — 40+ years of Unix tools

          
✓AI-native — LLMs trained on billions of CLI examples

        

      


      
      

        

          
🔌

          

            
MCP Approach

            
Model Context Protocol by Anthropic

          

        

        
The agent connects to a separate MCP server that advertises available tools via structured schemas. Each tool is a JSON definition the agent must interpret at runtime.


        
        

          
Tool schemas loaded into context

          

            {"name":"create_issue","params":{...}}

            {"name":"list_repos","params":{...}}

            {"name":"get_pull","params":{...}}

            {"name":"create_review","params":{...}}

            ... 39 more tools loaded

            ⚠ ~44,000 tokens consumed before any work
          

        


        

          
✗10-32x more expensive per operation

          
✗72% reliability — timeouts & failures

          
✗Massive token bloat — schemas eat 40-50% of context

          
⚠Security gaps — 30 CVEs in 60 days

          
⚠New to AI — models encounter schemas for the first time

What This Means for OpenClaw


      

        

          
CLI-first by design

          
OpenClaw's core agent (Pi) has just 4 built-in tools — including a shell. Through it, the agent can run any command. Skills teach the agent which commands to run for each task. No protocol overhead, no schema bloat.

        

        

          
MCP available when needed

          
When no CLI tool exists — like connecting to Notion or Stripe — OpenClaw supports MCP through community skills. But CLI is always preferred: faster, cheaper, and more reliable.

        

      


      

        
⚠ Without CLI-first tooling, OpenClaw's capabilities are severely limited. An MCP-only setup would consume 40-50% of the AI's context window on tool schemas alone — leaving less room for memory, skills, conversation history, and the actual task. Operations would cost 10-32x more, with a 28% failure rate. The agent's ability to chain complex multi-step workflows — the core value proposition — degrades significantly when every tool call burns thousands of tokens on protocol overhead.

      Industry trend: Perplexity, Cloudflare, and Y Combinator's CEO have all publicly moved away from MCP for agent workloads. The consensus is emerging: CLI for the inner loop (fast, local, frequent operations), MCP for the outer loop (enterprise auth, non-technical users, services without CLI tools).
    


  










  

    

      
06 Proof of Concept

      

        
        +
        
        OpenClaw
      

      
We built a CLI and a skill for our LLM evaluation platform qym (قيِّم) to explore what an OpenClaw integration would look like. The goal: let teams check eval runs, surface failures, and compare models straight from Slack — without opening a dashboard. Here's the proposed approach and early results.

    


    
    
Two ingredients to integrate any platform


    

      
      

        

          💻
          1. Build a CLI
          THE INTERFACE
        

        

          
Give your platform a command-line interface that the agent can call. Every command returns structured JSON.

          

            
qym CLI

            

              $ qym run list --json

              → [{run_id, task, model, success_rate}]

              $ qym run failed <run_id> --json

              → [{item, error, score: 0}]

              $ qym analyze run <run_id> --json

              → [{category, count, root_cause}]

              $ qym run compare <id1> <id2> --json

              → {metrics_diff, winner, delta}
            

          

          

            The rule: every command takes --json, errors go to stderr, exit codes are meaningful. That's it — the agent can now use your platform.
          

        

      


      
      

        

          ⚡
          2. Write a SKILL.md
          THE PLAYBOOK
        

        

          
Teach the agent when and how to use each command. Plain English — no code.

          

            
SKILL.md

            

              name: qym

              needs: qym CLI installed

              ---

              When asked about eval runs:

                → qym run list --json

              When a run has failures:

                → qym run failed <id> --json

                → qym analyze run <id> --json

              When comparing models:

                → qym run compare <a> <b> --json
            

          

          

            The skill is the bridge between "the tool exists" and "the agent knows how to use it." OpenClaw auto-loads it when qym is detected on the system.

06 Proactive

      
The Heartbeat

      
Every 30 minutes, OpenClaw wakes the agent to check a task list (HEARTBEAT.md). If nothing needs attention, it goes back to sleep. If something does — it acts.

    


    

      

        
💓


        

          

            

            Heartbeat: Off
          

        


        

          
        

      


      
      

        
HEARTBEAT.md

        

          # Every heartbeat (30 min)

          - [ ] Check email for urgent messages

          - [ ] Monitor Slack for @mentions

          - [ ] Check GitHub PRs needing review


          # Every morning (8 AM)

          - [ ] Summarize today's calendar

          - [ ] Send morning briefing via WhatsApp


          # Weekly (Monday 9 AM)

          - [ ] Compile weekly status report
